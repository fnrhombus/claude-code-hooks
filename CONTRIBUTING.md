# Contributing

Thanks for wanting to help! This project has two distinct types of contribution: **type updates** (automated) and **wrapper improvements** (human).

## Type updates — the pipeline handles these

`packages/core/src/types.ts` is **autogenerated** from `https://code.claude.com/docs/en/hooks.md`. Do not edit it by hand. The scheduled `scripts/dev-cycle` pipeline regenerates it whenever the upstream markdown changes and opens a PR automatically.

If you notice the types are wrong or out of date:

1. First check [`BLOCKERS.md`](./BLOCKERS.md) — the pipeline may have already tried and given up.
2. If nothing is listed, run the pipeline manually: `./scripts/dev-cycle` (Unix) or `scripts\dev-cycle.cmd` (Windows).
3. If that fails too, file an issue — don't paper over the regen by hand-editing `types.ts`.

## Wrapper / API improvements — PRs welcome

`packages/core/src/hook.ts`, `helpers.ts`, and `index.ts` are hand-written and evolve normally. If you want to add a helper, fix a bug, or improve ergonomics:

1. Fork and create a feature branch from `main`
2. Write a failing test in `packages/tests/src/`
3. Make it pass
4. Run `pnpm -r typecheck && pnpm -r build && pnpm -r test`
5. Open a PR against `main`

All PRs require CI green + at least one maintainer review before merge.

## Local setup

Requires Node 20+ and pnpm 10+. The project uses [mise](https://mise.jdx.dev/) for tool version management; `mise install` will fetch the right versions.

```bash
git clone https://github.com/fnrhombus/claude-code-hooks
cd claude-code-hooks
mise install        # or: corepack enable && nvm use 20
pnpm install
pnpm -r typecheck
pnpm --filter @fnrhombus/claude-code-hooks build
pnpm --filter claude-code-hooks-tests test
```

## Code style

- TypeScript strict mode, `exactOptionalPropertyTypes`, `noUncheckedIndexedAccess`, the works
- Prefer functional style; avoid classes except for the intentional `HookBlock` error
- No comments unless the *why* is non-obvious
- Test names should read like sentences: `it("allow() produces hookSpecificOutput with permissionDecision", …)`

## Release process

Releases are tag-triggered. Pushing a tag matching `v*.*.*` kicks off the `publish.yml` workflow, which publishes to npm via trusted publishing (OIDC — no long-lived tokens). Follow semver strictly:

- **patch**: bug fix in the wrapper, no new fields, no field removed
- **minor**: new optional fields from upstream, new helper functions
- **major**: upstream removed or renamed a field, wrapper API breaking change

The scheduled auto-sync PRs start as `patch` by default; if they need to bump `minor` or `major`, a maintainer edits the version before merging.
