# @fnrhombus/claude-code-hooks

Strongly-typed TypeScript wrapper for Claude Code hook entrypoints. This file is context for Claude (and future-me) when working in this repo.

## Architecture (what humans should know)

- **`packages/core/src/types.ts`** is autogenerated by the `regen-hook-types` skill from `https://code.claude.com/docs/en/hooks.md`. Never edit it by hand — edits get blown away on the next regen. Line 1 is a `// SOURCE_SHA256:` header that tracks the upstream markdown hash; the skill uses it as an idempotent fast-path.
- **`packages/core/src/hook.ts`** is the runtime dispatcher (the thing `runHook` lives in). This is hand-written. Edit freely.
- **`packages/core/src/helpers.ts`** is the shorthand-constructor layer (`allow()`, `deny()`, `updateInput()`, etc.). Hand-written.
- **`scripts/dev-cycle.ts`** is the autonomous type-sync pipeline — it runs the regen skill, opens an issue + branch + PR if upstream changed, runs a TDD loop with a retry budget, and merges on green. Cross-platform entrypoints are `scripts/dev-cycle` (bash shim) and `scripts/dev-cycle.cmd` (Windows shim), both delegating to `dev-cycle.ts` via `pnpm exec tsx`.
- **Tests** live in a separate workspace package (`packages/tests`) so `vitest` is never a dependency of the published package.

## Downstream: claude-code-pathfix

`claude-code-pathfix` consumes this package and **bundles it into its own tarball at build time** (via tsup + `noExternal: [/.*/]`). So:

- pathfix users never see `@fnrhombus/claude-code-hooks` in their installed deps — it's already inlined.
- Updating the wrapper does NOT automatically reach pathfix users. pathfix has to rebuild + republish.
- pathfix has **dependabot + auto-merge** configured: new wrapper versions open a PR in pathfix, CI runs, and on green pathfix squash-merges + auto-republishes. This chain is hands-off for **semver-compatible** updates only.

### ⚠️ Breaking changes — the dependabot chain does NOT auto-accept major bumps

When cutting a **major version** (breaking change), the `^0.x.y` / `^1.x.y` caret range in pathfix's `package.json` won't accept the new version. Dependabot will still open a PR, but CI might fail (missing API, signature change, etc.) or the range bump will be refused.

**Steps to ship a breaking wrapper change:**

1. Merge the breaking change on main here. Bump to the new major version (e.g. `0.1.x` → `0.2.0` is NOT considered breaking under npm's `^0.x.y` rules — `^0.1.1` does NOT include `0.2.0`. Even minor bumps in the `0.x` range break the caret.).
2. Wait for the CI publish workflow to push the new version to npm.
3. Go to `c:\dev\fnrhombus.claude-code-pathfix`, bump the `@fnrhombus/claude-code-hooks` dep range in `package.json` by hand (e.g. `^0.1.1` → `^0.2.0`), run `npm install`, run `npm run build` + smoke test, commit, push.
4. pathfix's auto-publish workflow will pick up the new pathfix version and ship it.

You'll know you're in this situation because the dependabot PR in pathfix will either (a) fail CI or (b) propose a range change that npm won't semver-match automatically.

### Pre-1.0 caveat

Under npm's default caret semantics, `^0.y.z` only allows patch updates (`0.y.*`). So **every minor bump while we're still in `0.x` is technically "breaking"** for dependabot and requires a manual range bump in pathfix. Once we hit `1.0.0`, normal caret rules kick in and only major bumps require manual intervention.

This is the main reason to get to `1.0.0` once the API feels stable.

## Tests

Tests are in `packages/tests/src/*.test.ts`, run via `pnpm --filter claude-code-hooks-tests test`. They cover the `dispatch` pure function and the `runHook` stdin/stdout integration. When upstream types change, the dispatcher tests are what catch the damage — if `dispatch` still routes correctly for every `HookEventName` variant, the tests pass.

`packages/tests` imports `@fnrhombus/claude-code-hooks` via the workspace symlink. That means `packages/core` must be **built** before tests can typecheck. CI and `scripts/dev-cycle.ts` both run `pnpm --filter @fnrhombus/claude-code-hooks build` before `pnpm -r typecheck`.

## Publishing

Tag-triggered via `.github/workflows/publish.yml`. Uses OIDC trusted publishing configured on npmjs.com — no NPM_TOKEN. The workflow uses `npx -y npm@latest publish` rather than `pnpm publish` because pnpm publish has inconsistent OIDC behavior with scoped packages (publishes signed provenance but gets a 404 on the PUT).

To release:

```bash
# Bump version in packages/core/package.json, then:
git commit -am "chore: bump to X.Y.Z"
git tag vX.Y.Z
git push && git push --tags
```

## Retry budget

Per user preference in `~/.claude/CLAUDE.md`: 5 attempts → research → 3 attempts → give up (file issue + BLOCKERS.md entry + assign to @fnrhombus). This repo has a `BLOCKERS.md` at the root. The autonomous `scripts/dev-cycle.ts` pipeline enforces this budget programmatically.
