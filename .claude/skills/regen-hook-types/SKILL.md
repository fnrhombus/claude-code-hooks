# regen-hook-types

Regenerates `packages/core/src/types.ts` from the upstream source at `https://code.claude.com/docs/en/hooks.md`.

The skill is **idempotent**: if the upstream markdown is byte-identical to the last regeneration, the skill short-circuits to a no-op and the output file is untouched. This fast path is **fully deterministic** ‚Äî no LLM reasoning happens: it's a file read, a string comparison, and an exit. Output does not depend on `--model` or `--effort`.

When the source *has* changed, the skill regenerates the file. Regeneration is not fully deterministic across models (LLM output varies in minor formatting), but the pinned procedure below minimizes drift.

## Procedure

You MUST follow these steps in order. Do not skip, reorder, or embellish.

### Step 1 ‚Äî Fetch the source and compute its hash

Run exactly:

```bash
curl -fsSL https://code.claude.com/docs/en/hooks.md -o /tmp/hooks-src.md \
  && sha256sum /tmp/hooks-src.md | awk '{print $1}'
```

Call the hex output `NEW_HASH`.

### Step 2 ‚Äî Read the first line of `packages/core/src/types.ts`

The generated file always starts with a header line of the exact form:

```
// SOURCE_SHA256: <64 lowercase hex chars>
```

Use `Read` with `limit: 1` to get just line 1. If the file doesn't exist or line 1 doesn't match that pattern, treat `OLD_HASH` as the empty string. Otherwise extract the hex after `SOURCE_SHA256: `.

### Step 3 ‚Äî If `NEW_HASH === OLD_HASH`, stop

Report **exactly** this sentence and nothing else:

> `hook types up to date (sha256: <NEW_HASH>)`

Do not touch the file. Do not fetch anything else. Do not add commentary, a summary, or next steps. End the turn.

This is the fully deterministic fast path. Every invocation on an unchanged source produces byte-identical output regardless of model, effort, or prior conversation.

### Step 4 ‚Äî Otherwise, regenerate

Read `/tmp/hooks-src.md` in full. From its content, produce `packages/core/src/types.ts` with the following invariants. Treat these as hard constraints, not suggestions.

**Line 1 is exactly** `// SOURCE_SHA256: <NEW_HASH>` ‚Äî lowercase hex, no trailing whitespace, followed by a newline.

**File structure, in this order:**

1. The `SOURCE_SHA256` header line (line 1).
2. A JSDoc block explaining what the file is, linking to the source URL, and warning that it's regenerated by the `regen-hook-types` skill.
3. Section `// === Enums ===` (wrapped in `// ==...==` banners): only multi-use string-literal unions (see "Inline single-use simple types" rule below). Declared in alphabetical order by type name. Do NOT include `HookEventName` here ‚Äî it is derived (see section 6).
4. Section: **Common input fields** ‚Äî `HookInputCommon` interface (with `hook_event_name: string` ‚Äî not `HookEventName`, to avoid circular refs), followed by the `HookEvent<E>` generic: `export type HookEvent<E extends string> = HookInputCommon & { hook_event_name: E };`
5. Section: **Tool input schemas** ‚Äî one `interface` per built-in tool. **Alphabetical by interface name.** Followed by `BuiltinToolInputMap`, `BuiltinToolName`, and the `ToolCall` discriminated union.
6. Section: **Per-event input types** ‚Äî one type per event, then the `HookInput` union, then `HookEventName` derived from it: `export type HookEventName = HookInput["hook_event_name"];`
7. Section: **Hook output schema** ‚Äî helper generics, then per-event output types, the `HookSpecificOutput` union, the `HookOutputMap` interface (event name ‚Üí output type, only for events that have specific output), then the `HookOutput` root interface.
8. Section: **Plugin/hook environment variables** ‚Äî `HookEnv` interface.

**DRY patterns ‚Äî use these to eliminate redundancy:**

*Input types:*
- `HookEvent<E>` generic: for events with ‚â§2 extra fields, use `export type FooInput = HookEvent<"Foo"> & { field: Type };` instead of a full interface. For events with zero extra fields, use `export type FooInput = HookEvent<"Foo">;`
- Events with 3+ extra fields: use a full `export interface FooInput extends HookInputCommon { ... }`.
- When two events share identical fields (differing only in `hook_event_name`), extract a shared `interface FooFields { ... }` (not exported) and intersect: `export type FooCreatedInput = HookEvent<"FooCreated"> & FooFields;`
- Tool-context events (`PreToolUse`, `PostToolUse`, etc.) use `ToolEvent<E>`: `type ToolEvent<E extends string> = HookEvent<E> & { permission_mode: PermissionMode; tool_use_id: string } & ToolCall;` Declare this helper right after `HookEvent<E>`. Events that override `tool_use_id` optionality (e.g., PermissionRequest) or narrow `permission_mode` (e.g., PermissionDenied) use `HookEvent<E> & { ... } & ToolCall` directly instead.

*Output types:*
- `HookEventOutput<E>` generic: `type HookEventOutput<E extends string> = { hookEventName: E };`
- `ContextOnlyOutput<E>`: `HookEventOutput<E> & { additionalContext?: string }` ‚Äî for events whose only output field is `additionalContext`.
- `SessionOutput<E>`: extends `ContextOnlyOutput<E>` with `sessionTitle?: string` ‚Äî for SessionStart and UserPromptSubmit.
- When two output types are identical except for the event name (e.g., Elicitation and ElicitationResult), extract a shared generic and instantiate it.
- Events with unique output fields: `export type FooHookOutput = HookEventOutput<"Foo"> & { ... };` or `ContextOnlyOutput<"Foo"> & { ... };`
- Events with NO specific output type should not have a `*HookOutput` type at all ‚Äî they return only universal `HookOutput` fields.

**Interface member ordering:** alphabetical. Discriminator fields (`hook_event_name`, `hookEventName`) are handled by the generics (`HookEvent<E>`, `HookEventOutput<E>`) and should not appear in the body of types that use those generics.

**Field typing rules:**

- Every field present in the source must appear. Mark as optional (`?:`) exactly those the source marks optional or explicitly describes as "only present in X context."
- Use the narrowest type the source supports: string-literal unions for enum-like fields, `number` for numeric, `boolean` for boolean, `string[]` for string arrays, `Record<string, unknown>` for arbitrary objects (never `any`, never bare `object`).
- Tool `tool_input` uses the `ToolCall` discriminated union (see `PreToolUseInput`, `PostToolUseInput`, etc.).
- Tool `tool_response` is `unknown`.

**Forbidden:**

- No `any`.
- No `@example` JSDoc tags, no inline usage snippets.
- **Rich JSDoc comments for IntelliSense.** Every exported type, interface, and field should have a JSDoc comment (`/** ... */`) derived from the source markdown. Include: what the type/field represents, when it's present, what values mean, constraints, units, which events it applies to, and any caveats or gotchas mentioned in the source. The goal is that a consumer hovering over any symbol in their IDE gets all the information they'd find in the docs without leaving their editor. Do NOT invent information ‚Äî only document what the source markdown states. Section banners (`// === ... ===`) are still fine as structural separators.
- No `export default`, no classes, no runtime code. Only `export type` and `export interface`.
- No imports. This file must be self-contained.
- **Inline single-use simple types.** If a string-literal union type is referenced exactly once in the file (as a field type in one interface), do not declare it as a standalone `export type`. Instead, inline the union directly at the usage site. Types referenced two or more times, or that are interfaces/complex shapes, remain standalone. Example: if `SessionSource` is only used as `SessionStartInput.source`, write `source: "startup" | "resume" | "clear" | "compact";` directly in the interface instead of declaring `export type SessionSource = ...` separately.

**File ending:** exactly one trailing newline.

### Step 5 ‚Äî Verify and report

After writing, re-read line 1 and confirm it matches `// SOURCE_SHA256: <NEW_HASH>` exactly. If it doesn't, the write failed; fix and retry.

Report **exactly**: `regenerated packages/core/src/types.ts (sha256: <NEW_HASH>)`. No further commentary.

### Step 6 ‚Äî Do NOT run tests, build, commit, or push

This skill's only job is to write the types file. Downstream handling ‚Äî running tests, fixing wrapper code that references removed fields, committing, opening a PR ‚Äî is the responsibility of the caller (typically `scripts/dev-cycle.sh`).

## Determinism guarantees

| Path | Trigger | Deterministic across `--model` / `--effort`? |
|---|---|---|
| Fast (Steps 1‚Äì3) | Source unchanged | **Yes, fully.** No LLM reasoning ‚Äî just `curl`, `sha256sum`, one-line read, string compare. |
| Slow (Step 4)    | Source changed  | **No, only approximately.** The pinned ordering and typing rules above minimize model-to-model drift, but expect small diffs (whitespace, comment phrasing, edge-case member ordering). |

For true determinism on the slow path too, replace Step 4 with a committed `scripts/regen-hook-types.js` Node script that parses `hooks.md` with a regex pipeline and emits the `.ts` directly. The skill's Step 4 would then collapse to "run `node scripts/regen-hook-types.js`." That's strictly better but requires writing and maintaining the parser ‚Äî deferred until the upstream doc format stabilizes.
