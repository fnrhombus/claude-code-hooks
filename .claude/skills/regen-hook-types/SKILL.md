# regen-hook-types

Regenerates `packages/core/src/types.ts` from the upstream source at `https://code.claude.com/docs/en/hooks.md`.

The skill is **idempotent**: if the upstream markdown is byte-identical to the last regeneration, the skill short-circuits to a no-op and the output file is untouched. This fast path is **fully deterministic** â€” no LLM reasoning happens: it's a file read, a string comparison, and an exit. Output does not depend on `--model` or `--effort`.

When the source *has* changed, the skill regenerates the file. Regeneration is not fully deterministic across models (LLM output varies in minor formatting), but the pinned procedure below minimizes drift.

## Procedure

You MUST follow these steps in order. Do not skip, reorder, or embellish.

### Step 1 â€” Fetch the source and compute its hash

Run exactly:

```bash
curl -fsSL https://code.claude.com/docs/en/hooks.md -o /tmp/hooks-src.md \
  && sha256sum /tmp/hooks-src.md | awk '{print $1}'
```

Call the hex output `NEW_HASH`.

### Step 2 â€” Read the first line of `packages/core/src/types.ts`

The generated file always starts with a header line of the exact form:

```
// SOURCE_SHA256: <64 lowercase hex chars>
```

Use `Read` with `limit: 1` to get just line 1. If the file doesn't exist or line 1 doesn't match that pattern, treat `OLD_HASH` as the empty string. Otherwise extract the hex after `SOURCE_SHA256: `.

### Step 3 â€” If `NEW_HASH === OLD_HASH`, stop

Report **exactly** this sentence and nothing else:

> `hook types up to date (sha256: <NEW_HASH>)`

Do not touch the file. Do not fetch anything else. Do not add commentary, a summary, or next steps. End the turn.

This is the fully deterministic fast path. Every invocation on an unchanged source produces byte-identical output regardless of model, effort, or prior conversation.

### Step 4 â€” Otherwise, regenerate

Read `/tmp/hooks-src.md` in full. From its content, produce `packages/core/src/types.ts` with the following invariants. Treat these as hard constraints, not suggestions.

**Line 1 is exactly** `// SOURCE_SHA256: <NEW_HASH>` â€” lowercase hex, no trailing whitespace, followed by a newline.

**File structure, in this order:**

1. The `SOURCE_SHA256` header line (line 1).
2. A JSDoc block explaining what the file is, linking to the source URL, and warning that it's regenerated by the `regen-hook-types` skill.
3. Section `// === Enums ===` (wrapped in `// ==...==` banners): every string-literal union from the source (`PermissionMode`, `HookEventName`, `SessionSource`, `SessionEndReason`, `NotificationType`, `StopErrorType`, `MemoryType`, `LoadReason`, `ConfigSource`, `CompactTrigger`, and any others that appear). Declared in alphabetical order by type name.
4. Section: **Common input fields** â€” `HookInputCommon` interface.
5. Section: **Tool input schemas** â€” one `interface` per built-in tool (`BashToolInput`, `EditToolInput`, `GlobToolInput`, `GrepToolInput`, `ReadToolInput`, `WebFetchToolInput`, `WebSearchToolInput`, `WriteToolInput`, `AgentToolInput`, `AskUserQuestionInput`, `ExitPlanModeInput`, plus any new tools). **Alphabetical by interface name.** Followed by `BuiltinToolInputMap`, `BuiltinToolName`, and the `ToolCall` discriminated union.
6. Section: **Per-event input types** â€” one interface per value of `HookEventName`, declared in the same order the events appear in the `HookEventName` union. Followed by the `HookInput` union (listing every per-event interface in the same order).
7. Section: **Hook output schema** â€” `PermissionDecision` type alias, one `*HookOutput` interface per event that has hook-specific output fields (in `HookEventName` order), the `HookSpecificOutput` union, then the `HookOutput` root interface.
8. Section: **Plugin/hook environment variables** â€” `HookEnv` interface.

**Interface member ordering:** alphabetical, with one exception â€” the discriminator field (`hook_event_name` for inputs, `hookEventName` for outputs) comes first when present.

**Field typing rules:**

- Every field present in the source must appear. Mark as optional (`?:`) exactly those the source marks optional or explicitly describes as "only present in X context."
- Use the narrowest type the source supports: string-literal unions for enum-like fields, `number` for numeric, `boolean` for boolean, `string[]` for string arrays, `Record<string, unknown>` for arbitrary objects (never `any`, never bare `object`).
- Tool `tool_input` uses the `ToolCall` discriminated union (see `PreToolUseInput`, `PostToolUseInput`, etc.).
- Tool `tool_response` is `unknown`.

**Forbidden:**

- No `any`.
- No examples, no `@example` JSDoc tags, no inline usage snippets.
- No comments beyond the header, the section banners, and one-line field clarifications where the source explicitly defines units or constraints ("Milliseconds.", "Only set when...", etc.).
- No `export default`, no classes, no runtime code. Only `export type` and `export interface`.
- No imports. This file must be self-contained.

**File ending:** exactly one trailing newline.

### Step 5 â€” Verify and report

After writing, re-read line 1 and confirm it matches `// SOURCE_SHA256: <NEW_HASH>` exactly. If it doesn't, the write failed; fix and retry.

Report **exactly**: `regenerated packages/core/src/types.ts (sha256: <NEW_HASH>)`. No further commentary.

### Step 6 â€” Do NOT run tests, build, commit, or push

This skill's only job is to write the types file. Downstream handling â€” running tests, fixing wrapper code that references removed fields, committing, opening a PR â€” is the responsibility of the caller (typically `scripts/dev-cycle.sh`).

## Determinism guarantees

| Path | Trigger | Deterministic across `--model` / `--effort`? |
|---|---|---|
| Fast (Steps 1â€“3) | Source unchanged | **Yes, fully.** No LLM reasoning â€” just `curl`, `sha256sum`, one-line read, string compare. |
| Slow (Step 4)    | Source changed  | **No, only approximately.** The pinned ordering and typing rules above minimize model-to-model drift, but expect small diffs (whitespace, comment phrasing, edge-case member ordering). |

For true determinism on the slow path too, replace Step 4 with a committed `scripts/regen-hook-types.js` Node script that parses `hooks.md` with a regex pipeline and emits the `.ts` directly. The skill's Step 4 would then collapse to "run `node scripts/regen-hook-types.js`." That's strictly better but requires writing and maintaining the parser â€” deferred until the upstream doc format stabilizes.
